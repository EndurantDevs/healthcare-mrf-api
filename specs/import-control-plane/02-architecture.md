# 02 Architecture

## Purpose

Define component boundaries for a distributed import system where one central control service coordinates many independent MRF serving/import nodes.

## Components

### `import-control`

New central service and database. It owns:

- MRF source catalog.
- Source seed review.
- Discovered plans/files.
- Schedules.
- Client subscriptions.
- File-level import coalescing.
- Node registry and health.
- PTG placement and route index.
- Run mirror and cross-engine history.
- Quotas, locks, fair queueing, and audit.

It does not directly scan MRF files or serve pricing queries.

### `healthcare-mrf-api`

Execution and serving engine. Each instance owns:

- Its own PostgreSQL database.
- Its own Redis/ARQ workers.
- Its own PTG artifact store.
- Local base datasets.
- Local PTG snapshots.
- Engine `/control/v1` API.

Each node remains autonomous. Engine run truth lives locally in `import_run`; `import-control` mirrors it from async engine-pushed status events. Normal run/status reads use the central mirror and do not synchronously poll importer nodes; manual sync is an operator repair path.

### `api-layer`

Public gateway and client enforcement point. It owns:

- API key/JWT validation.
- Bundle and entitlement enforcement.
- Client-safe import APIs.
- PTG route lookup and upstream selection.
- Public coverage API proxying.

Current code assumes one healthcare upstream. Multi-node PTG routing is a new capability.

### `api-app`

Human UI. It owns:

- Admin import dashboard.
- Client import dashboard.
- Public coverage pages.
- Internal API proxy/adapters to `import-control`.

### `api-mcp`

Agent tools. Client-scoped tools route through `api-layer`; admin tools may call `import-control` directly.

## Distributed Node Model

- One control node may coordinate many serving/import nodes.
- Every serving node may have its own PostgreSQL, Redis, API port, worker pool, and artifact root.
- Base imports are normally replicated to all serving nodes.
- PTG imports are placed selectively by source file/content version.
- `import-control` records where each source file/content version was imported.
- `api-layer` uses the route index to select the serving node for PTG requests.

## Data Ownership

- Catalog truth: `import-control`.
- Engine run truth: local engine DB.
- Schedule/config truth: `import-control` DB.
- Node health truth: latest node heartbeat plus control polling.
- PTG serving truth: local node snapshot tables and manifests.
- Public API route truth: `ptg_route_index` generated by `import-control`.

## Failure Behavior

- If `import-control` is down, existing pricing APIs continue using cached/default routes until route cache expires.
- If a serving node is down, PTG requests route to replicated candidates only; no unbounded fanout.
- If no node has a requested PTG dataset, client-facing APIs return a clear not-imported response.
- If the scheduler is down, no new scheduled imports start, but existing engine runs continue.
- If an engine stops heartbeating, `import-control` marks its runs stale and node unhealthy.
- Live progress keys expire in Redis; terminal run mirror history is purged by retention policy.

## Acceptance Criteria

- The system diagram can be implemented without circular dependencies.
- Every component has one clear owner for write operations.
- Cross-node PTG routing does not require direct DB access from `api-layer`.
